# Docker fnOS ARM64 Release Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Package Media Forge as a single Docker application image for fnOS ARM64 that serves the built frontend through FastAPI and persists runtime files through one mounted `data` directory.

**Architecture:** FastAPI remains the single runtime process and owns both API routes and frontend static serving. Docker uses a multi-stage build: Node builds `frontend/dist`, Python installs backend dependencies, and the final image copies the frontend build into `backend/app/static`. Makefile targets wrap local builds, ARM64 Buildx packaging, and tar export into `output/`.

**Tech Stack:** Python 3.12, FastAPI 0.115, Uvicorn, React 19, Vite 8, TypeScript 6, Docker Buildx, GNU Make, pytest.

## Global Constraints

- Build one application image, not a Compose stack.
- Do not include PostgreSQL or Redis in the image.
- Do not run database migrations automatically on container startup.
- Expose only container port `18642`.
- Use `/app/data` as the only runtime volume.
- `APP_CONFIG_DIR=/app/data/configs`.
- `LOG_DIR=/app/data/logs`.
- `make docker-build-arm64` must automatically write `output/media-forge-linux-arm64.tar`.
- The repository `data/` directory must not be copied into the image or exported tar.
- The repository `output/` directory must not be copied into the image build context.
- Preserve all current `/api/*` paths and frontend routes.
- Do not change authentication, initialization, crawler, storage, or media processing behavior.
- Do not stage unrelated existing changes in `frontend/src/components/BaseListPage/index.tsx`, `frontend/src/pages/content/movies/components/MovieTable.tsx`, or existing untracked plan files.

```python
def configure_frontend(
    target_app: FastAPI,
    static_dir: Path = FRONTEND_STATIC_DIR,
) -> None:
    """Serve built frontend assets when the Vite dist output is present."""
    index_file = static_dir / FRONTEND_INDEX_FILE
    assets_dir = static_dir / "assets"
    if not index_file.exists():
        return

    if assets_dir.exists():
        target_app.mount(
            "/assets",
            StaticFiles(directory=assets_dir),
            name="frontend-assets",
        )

    @target_app.get("/", include_in_schema=False)
    def frontend_root() -> FileResponse:
        return FileResponse(index_file)

    @target_app.get("/{full_path:path}", include_in_schema=False)
    def frontend_fallback(full_path: str) -> Response:
        if full_path == "api" or full_path.startswith("api/"):
            return Response(status_code=404, media_type="application/json", content='{"detail":"Not Found"}')
        requested_file = static_dir / full_path
        if requested_file.is_file():
            return FileResponse(requested_file)
        return FileResponse(index_file)
```

Call it after all routers are included:


- [ ] **Step 1: Create `Makefile`**

Create root `Makefile`:

```makefile
APP_NAME ?= media-forge
IMAGE ?= $(APP_NAME)
IMAGE_TAG ?= latest
ARM64_TAG ?= arm64
CONTAINER_NAME ?= media-forge
HOST_PORT ?= 18642
CONTAINER_PORT ?= 18642
DATA_DIR ?= $(CURDIR)/data
OUTPUT_DIR ?= $(CURDIR)/output
ARM64_TAR ?= $(OUTPUT_DIR)/media-forge-linux-arm64.tar

.PHONY: frontend-build docker-build docker-build-arm64 docker-save-arm64 docker-run docker-stop output-dir

frontend-build:
	cd frontend && npm ci && npm run build

docker-build:
	docker build -t $(IMAGE):$(IMAGE_TAG) .

output-dir:
	mkdir -p $(OUTPUT_DIR)

docker-build-arm64: output-dir
	docker buildx build --platform linux/arm64 --load -t $(IMAGE):$(ARM64_TAG) .
	docker save $(IMAGE):$(ARM64_TAG) -o $(ARM64_TAR)
	@echo "ARM64 image tar written to $(ARM64_TAR)"

docker-save-arm64: output-dir
	docker save $(IMAGE):$(ARM64_TAG) -o $(ARM64_TAR)
	@echo "ARM64 image tar written to $(ARM64_TAR)"

docker-run: docker-build
	mkdir -p $(DATA_DIR)
	docker rm -f $(CONTAINER_NAME) >/dev/null 2>&1 || true
	docker run -d \
		--name $(CONTAINER_NAME) \
		-p $(HOST_PORT):$(CONTAINER_PORT) \
		-v $(DATA_DIR):/app/data \
		$(IMAGE):$(IMAGE_TAG)

docker-stop:
	docker rm -f $(CONTAINER_NAME) >/dev/null 2>&1 || true
```

````markdown
# Docker fnOS ARM64 Deployment

Media Forge is packaged as a single application image for fnOS ARM64. The image serves the React frontend and FastAPI backend on one port.

## Build ARM64 Package

Run from the repository root:

```bash
make docker-build-arm64
```

The build writes:

```text
output/media-forge-linux-arm64.tar
```

Upload this tar file to fnOS or load it on an ARM64 Docker host:

```bash
docker load -i output/media-forge-linux-arm64.tar
```

## Runtime Requirements

The image does not include PostgreSQL or Redis. Provide external PostgreSQL and Redis services that the container can reach from fnOS.

Do not use `localhost` or `127.0.0.1` for PostgreSQL or Redis in the `/init` page unless the service runs inside the same container. In Docker, those addresses point to the Media Forge container itself.

Use one of these host values instead:

- PostgreSQL or Redis on the fnOS host: use the fnOS LAN IP, for example `192.168.1.20`.
- PostgreSQL or Redis on Docker Desktop host: use `host.docker.internal`.
- PostgreSQL or Redis in another container: put both containers on the same Docker network and use the database container name, for example `postgres`.
- PostgreSQL or Redis on another machine: use that machine's reachable LAN IP or DNS name.

Only one host directory is required for persistence:

```text
/path/to/media-forge-data
```

It is mounted to:

```text
/app/data
```

Media Forge writes runtime files under:

```text
/app/data/configs/database.conf
/app/data/configs/redis.conf
/app/data/configs/storage.conf
/app/data/logs/
```

The repository `data/` directory is not packaged into the Docker image or ARM64 tar. The `output/` directory stores release tar files only and is not copied into the image.

## Run Container

```bash
docker run -d \
  --name media-forge \
  -p 18642:18642 \
  -v /path/to/media-forge-data:/app/data \
  media-forge:arm64
```

Open:

```text
http://<fnos-host>:18642
```

## First-Time Initialization

On first startup, if `database.conf` and `redis.conf` do not exist, Media Forge routes the browser to `/init`.

When initialization settings are saved, the backend:

- validates PostgreSQL;
- creates the target database when missing;
- creates application tables;
- repairs incompatible empty crawler task tables;
- creates the default admin user `admin/admin123` when missing;
- validates Redis;
- writes runtime config under `/app/data/configs`;
- reloads runtime configuration.

The container does not automatically run Alembic migrations on startup. Future schema migrations must be run explicitly.

## Local Smoke Check

```bash
curl -i http://127.0.0.1:18642/
curl -i http://127.0.0.1:18642/api/init/config
curl -i http://127.0.0.1:18642/login
```
````
